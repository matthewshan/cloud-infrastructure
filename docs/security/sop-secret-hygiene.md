# SOP: Secret Hygiene for Infrastructure Development

Applies to all Terraform modules in `matthewshan/cloud-infrastructure`.
This is a **public repository** — any secret committed here is immediately
public and must be treated as compromised.

---

## 1. Pre-commit controls

### 1.1 `.gitignore` must be the first commit in every module

Before writing any `.tf` file, ensure the repo-root `.gitignore` covers:

```gitignore
*.tfstate
*.tfstate.*
.terraform/
*.tfvars
!*.tfvars.example
crash.log
crash.*.log
override.tf
*_override.tf
```

Never add exceptions (`git add -f`) for these patterns.

### 1.2 Mark all sensitive outputs in Terraform

Every `output` that contains a credential, key, token, or private value
**must** carry `sensitive = true`:

```hcl
output "service_account_key_base64" {
  value     = google_service_account_key.daily_briefing.private_key
  sensitive = true
}
```

Terraform will redact it in plan/apply output. Without this flag it prints
in plaintext in HCP Terraform run logs, which are visible to anyone with
workspace access.

### 1.3 Never use `terraform output` in CI without `-json` + masking

If a pipeline step must read an output value:

```bash
# Safe — value never appears in raw log text
VALUE=$(terraform output -raw -no-color my_secret | base64)
```

Prefer passing secrets directly between HCP Terraform workspaces or writing
them to a secrets manager (GCP Secret Manager, GitHub Actions secrets) rather
than echoing them in scripts.

## 2. State file handling

Terraform state is the highest-risk artifact in this repo. State files:

- Store all resource attribute values, including credentials generated by
  `google_service_account_key` and similar resources
- Are **not encrypted by default** when stored locally
- Are stored **encrypted at rest** in HCP Terraform

### Rules

| Rule | Enforcement |
|---|---|
| Never commit `*.tfstate` | `.gitignore` + GitHub secret scanning |
| Never store state locally long-term | Always use HCP Terraform `cloud {}` backend |
| Never copy state into a script or log | Read specific values with `terraform output -raw` |
| Rotate any key that appeared in a committed state file | Manual; see §5 |

### Verifying state is in HCP Terraform (not local)

```bash
# Should return "remote" — if it returns "local", the backend is misconfigured
terraform providers lock
terraform workspace show
```

Or check that `main.tf` has a `cloud {}` block and no local `terraform.tfstate`
file exists in the module directory.

## 3. HCP Terraform variable configuration

All sensitive values must be stored as **HCP Terraform workspace variables**,
not in `.tfvars` files or environment variables on a developer machine.

### Checklist for each workspace

- [ ] `project_id` — Terraform variable, not sensitive
- [ ] `GOOGLE_CREDENTIALS` — Environment variable, **marked sensitive**
- [ ] Any API key, token, or private key — Environment variable, **marked sensitive**

**Marking a variable sensitive in HCP Terraform** makes it write-only: it is
never displayed in the UI, never printed in run logs, and cannot be read back
via the API. Do this for every credential at creation time — it cannot be
undone without deleting and recreating the variable.

## 4. GitHub Actions log safety

If GitHub Actions workflows are added to this repo in future, follow these
rules to prevent secrets from appearing in job logs.

### 4.1 Register every secret before use

All secrets must be registered in **Settings → Secrets and variables →
Actions** before being referenced in a workflow. GitHub automatically masks
registered secret values in logs.

```yaml
- name: Authenticate to GCP
  env:
    GOOGLE_CREDENTIALS: ${{ secrets.GOOGLE_CREDENTIALS }}
```

### 4.2 Never echo, print, or expand secrets directly

```yaml
# BAD — prints the secret in plaintext
- run: echo "${{ secrets.MY_SECRET }}"

# BAD — shell expansion can leak if set -x is active
- run: |
    set -x
    do_something "${{ secrets.MY_SECRET }}"

# GOOD — pass via environment variable; GitHub masks it
- run: do_something "$MY_SECRET"
  env:
    MY_SECRET: ${{ secrets.MY_SECRET }}
```

### 4.3 Disable debug logging in production workflows

`ACTIONS_STEP_DEBUG=true` causes GitHub Actions to print all environment
variables, including secrets. Never set this in a workflow file; only enable
it temporarily via a repo secret for a specific debug run.

### 4.4 Prefer HCP Terraform VCS integration over Actions for Terraform runs

HCP Terraform's native VCS integration runs `plan` and `apply` inside HCP
Terraform's own environment where:

- Sensitive workspace variables are never written to run logs
- Plan output redacts `sensitive = true` outputs automatically
- No credentials are stored in GitHub at all

Use GitHub Actions only for non-Terraform steps (linting, policy checks, etc.).

## 5. Incident response — key exposed in git

If a credential, private key, or state file is detected in git history:

1. **Rotate the credential immediately** — assume it is already compromised.
   Do not wait to assess blast radius first.

   - GCP service account key: IAM Console → Service Accounts → Keys → Delete
     the exposed key ID, then create a new key
   - Any other secret: follow the issuing service's revocation process

2. **Revoke the old credential** in the issuing service before removing it
   from git history — removing from history does not help if the secret is
   still valid.

3. **Remove from git history** using `git filter-repo` (preferred over
   `filter-branch`):

   ```bash
   # Install: pip install git-filter-repo
   git filter-repo --path path/to/secret/file --invert-paths
   git push origin --force --all
   git push origin --force --tags
   ```

4. **Invalidate any cached copies** — GitHub caches content for some time
   after a force-push. Open a GitHub Support ticket to purge the cache for
   a public repo.

5. **Audit access logs** in GCP (Cloud Audit Logs) for the exposed credential
   covering the window between first commit and rotation.

6. **Update this SOP** if the incident revealed a gap in controls.

## 6. Pre-merge checklist

Before merging any PR that adds or modifies Terraform code:

- [ ] No `*.tfstate` files in the diff
- [ ] No `.terraform/` directory contents in the diff
- [ ] No hardcoded project IDs, key IDs, or credential values in `.tf` files
- [ ] All new `output` blocks containing credentials have `sensitive = true`
- [ ] New variables with secret values are documented in `.tfvars.example`
      with placeholder values only
- [ ] HCP Terraform speculative plan shows no unintended resource changes
