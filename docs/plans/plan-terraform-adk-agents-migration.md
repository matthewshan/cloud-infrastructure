# Migration Plan: adk-playground terraform ŌåÆ cloud-infrastructure

Migrate the Google Calendar service account Terraform module from
`matthewshan/adk-playground` into this repo (`matthewshan/cloud-infrastructure`)
under `terraform-adk-agents/`, switching from OpenTofu with local state to
HashiCorp Terraform with HCP Terraform (free tier) as the remote backend.

---

## Context

| | Source | Destination |
|---|---|---|
| **Repo** | `matthewshan/adk-playground` | `matthewshan/cloud-infrastructure` |
| **Path** | `terraform/modules/google-calendar-sa/` | `terraform-adk-agents/` |
| **Toolchain** | OpenTofu Ōēź 1.6 | HashiCorp Terraform Ōēź 1.6 |
| **State backend** | Local `terraform.tfstate` | HCP Terraform (cloud) |

---

## Phase 1 ŌĆö Audit for secret exposure

**Do this first**, before committing anything.

The source `terraform.tfstate` contains a GCP service account private key.
Check whether it was ever committed to `adk-playground`:

```powershell
git -C <path-to-adk-playground> log --all --oneline -- terraform/modules/google-calendar-sa/terraform.tfstate
```

**If any commits are returned:** the private key has been exposed in git history.
Rotate it immediately in [GCP IAM Console](https://console.cloud.google.com/iam-admin/serviceaccounts)
before continuing.

## Phase 3 ŌĆö Copy and split the Terraform code

Copy only the source code ŌĆö not state, not the provider cache:

| Copy from `adk-playground` | To `cloud-infrastructure` |
|---|---|
| `terraform/modules/google-calendar-sa/main.tf` | `terraform-adk-agents/` (then split) |

**Do NOT copy:**
- `terraform.tfstate` ŌĆö migrated separately via `terraform init`, never committed
- `.terraform/` ŌĆö regenerated by `terraform init`
- `.terraform.lock.hcl` ŌĆö must be regenerated by HashiCorp Terraform; the OpenTofu-generated lock file references `registry.opentofu.org` and will cause errors

Split the monolithic `main.tf` into three files:

- `main.tf` ŌĆö `terraform {}` block, `provider "google"`, and all `resource` blocks
- `variables.tf` ŌĆö all `variable` blocks
- `outputs.tf` ŌĆö all `output` blocks

**Update the `terraform {}` block** in `main.tf` ŌĆö replace the no-backend comment
with the HCP Terraform `cloud` block:

```hcl
terraform {
  required_version = ">= 1.6"

  cloud {
    organization = "<your-hcp-org-name>"
    workspaces {
      name = "google-calendar-sa"
    }
  }

  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}
```

## Phase 4 ŌĆö Migrate state from OpenTofu local ŌåÆ HCP Terraform

The existing state records the provider source as
`registry.opentofu.org/hashicorp/google`. HashiCorp Terraform uses
`registry.terraform.io`. The state must be patched before Terraform will
accept it.

### 4a. Copy the state file locally (do NOT commit it)

```powershell
Copy-Item `
  "<path-to-adk-playground>\terraform\modules\google-calendar-sa\terraform.tfstate" `
  "terraform-adk-agents\terraform.tfstate"
```

### 4b. Patch the provider registry URL

In `terraform-adk-agents/terraform.tfstate`, find-replace:

- **Find:** `registry.opentofu.org/hashicorp/google`
- **Replace:** `registry.terraform.io/hashicorp/google`

### 4c. Authenticate to HCP Terraform

```bash
terraform login
```

### 4d. Initialize ŌĆö Terraform will offer to migrate state

```bash
cd terraform-adk-agents
terraform init
```

Terraform detects the local `terraform.tfstate` next to the `cloud {}` backend
and prompts:

```
Do you want to copy existing state to the new backend?
  Enter a value: yes
```

State is now stored in HCP Terraform. The local file is no longer needed.

### 4e. Verify, then delete the local state file

```bash
terraform state list   # expect: 3 resources
terraform plan         # expect: No changes
```

```powershell
Remove-Item terraform-adk-agents\terraform.tfstate
```

## Phase 5 ŌĆö Configure variables in HCP Terraform

Never commit real variable values to the public repo. Set them in the HCP
Terraform workspace UI under **Variables**:

| Variable | Sensitive | Category | Notes |
|---|---|---|---|
| `project_id` | No | Terraform | GCP project ID |
| `GOOGLE_CREDENTIALS` | **Yes** | Environment | Full JSON of a GCP service account key with permission to manage IAM in the project |

`GOOGLE_CREDENTIALS` is the Google provider's standard env var for CI
authentication ŌĆö no key file on disk or in code required.

Add a `terraform.tfvars.example` in the module directory to document what
variables callers must supply:

```hcl
# Copy to terraform.tfvars (gitignored) for local runs.
# In HCP Terraform, set these as workspace variables instead.
project_id = "your-gcp-project-id"
# region = "us-central1"  # optional, defaults to us-central1
```

## Phase 6 ŌĆö CI/CD via GitHub Actions

The HCP Terraform workspace stays **CLI-driven** (no VCS link). GitHub Actions
triggers runs via the HCP Terraform API using `hashicorp/tfc-workflows-github`.

**Why not `hashicorp/setup-terraform` + `terraform plan` directly?**  
This repo is public. Direct `terraform plan/apply` streams full output ŌĆö
including sensitive resource attributes ŌĆö into public GitHub Actions logs.
The `tfc-workflows-github` actions trigger runs via the HCP Terraform API
instead; the full plan/apply output stays in the HCP Terraform UI (private)
and only a pass/fail status is reported back to Actions.

### 6a. Mark sensitive outputs in `outputs.tf`

Any output that contains a secret must use `sensitive = true` to prevent
Terraform from ever printing the value in plain text:

```hcl
output "service_account_key" {
  value     = google_service_account_key.calendar_sa_key.private_key
  sensitive = true
}
```

### 6b. Create a HCP Terraform API token for CI

HCP Terraform ŌåÆ User Settings ŌåÆ **Tokens** ŌåÆ Create an API token.  
Add it as a GitHub Actions secret named `TF_API_TOKEN` in  
`matthewshan/cloud-infrastructure` ŌåÆ Settings ŌåÆ Secrets ŌåÆ Actions.

### 6c. Create the workflow file

Create `.github/workflows/terraform-adk-agents.yml`:

```yaml
name: Terraform ŌĆö adk-agents

on:
  push:
    branches: [main]
    paths: [terraform-adk-agents/**]
  pull_request:
    branches: [main]
    paths: [terraform-adk-agents/**]

env:
  TF_CLOUD_ORGANIZATION: "<your-hcp-org-name>"
  TF_WORKSPACE: "google-calendar-sa"
  TF_DIRECTORY: "terraform-adk-agents"

jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/tfc-workflows-github/actions/upload-configuration@v1
        id: upload
        with:
          workspace: ${{ env.TF_WORKSPACE }}
          directory: ${{ env.TF_DIRECTORY }}
        env:
          TF_API_TOKEN: ${{ secrets.TF_API_TOKEN }}

      - uses: hashicorp/tfc-workflows-github/actions/create-run@v1
        id: run
        with:
          workspace: ${{ env.TF_WORKSPACE }}
          configuration_version: ${{ steps.upload.outputs.configuration_version_id }}
          plan_only: ${{ github.event_name == 'pull_request' }}
        env:
          TF_API_TOKEN: ${{ secrets.TF_API_TOKEN }}

      - uses: hashicorp/tfc-workflows-github/actions/apply-run@v1
        if: github.event_name == 'push' && github.ref == 'refs/heads/main'
        with:
          run: ${{ steps.run.outputs.run_id }}
          comment: "Triggered by merge to main"
        env:
          TF_API_TOKEN: ${{ secrets.TF_API_TOKEN }}
```

The `paths` filter ensures the workflow only fires when files inside
`terraform-adk-agents/` change ŌĆö adding future modules won't trigger it.
