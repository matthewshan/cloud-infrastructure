# Migration Plan: adk-playground terraform → cloud-infrastructure

Migrate the Google Calendar service account Terraform module from
`matthewshan/adk-playground` into this repo (`matthewshan/cloud-infrastructure`)
under `terraform-adk-agents/`, switching from OpenTofu with local state to
HashiCorp Terraform with HCP Terraform (free tier) as the remote backend.

---

## Context

| | Source | Destination |
|---|---|---|
| **Repo** | `matthewshan/adk-playground` | `matthewshan/cloud-infrastructure` |
| **Path** | `terraform/modules/google-calendar-sa/` | `terraform-adk-agents/` |
| **Toolchain** | OpenTofu ≥ 1.6 | HashiCorp Terraform ≥ 1.6 |
| **State backend** | Local `terraform.tfstate` | HCP Terraform (cloud) |

---

## Prerequisites

- [ ] HCP Terraform account at [app.terraform.io](https://app.terraform.io)
- [ ] HCP Terraform organization created (record the org name)
- [ ] HCP Terraform workspace created: `google-calendar-sa`
- [ ] HashiCorp `terraform` CLI installed (not OpenTofu)
- [ ] GCP credentials available locally for the one-time migration run

## Phase 1 — Audit for secret exposure

**Do this first**, before committing anything.

The source `terraform.tfstate` contains a GCP service account private key.
Check whether it was ever committed to `adk-playground`:

```powershell
git -C <path-to-adk-playground> log --all --oneline -- terraform/modules/google-calendar-sa/terraform.tfstate
```

**If any commits are returned:** the private key has been exposed in git history.
Rotate it immediately in [GCP IAM Console](https://console.cloud.google.com/iam-admin/serviceaccounts)
before continuing.

## Phase 3 — Copy and split the Terraform code

Copy only the source code — not state, not the provider cache:

| Copy from `adk-playground` | To `cloud-infrastructure` |
|---|---|
| `terraform/modules/google-calendar-sa/main.tf` | `terraform-adk-agents/` (then split) |

**Do NOT copy:**
- `terraform.tfstate` — migrated separately via `terraform init`, never committed
- `.terraform/` — regenerated by `terraform init`
- `.terraform.lock.hcl` — must be regenerated by HashiCorp Terraform; the OpenTofu-generated lock file references `registry.opentofu.org` and will cause errors

Split the monolithic `main.tf` into three files:

- `main.tf` — `terraform {}` block, `provider "google"`, and all `resource` blocks
- `variables.tf` — all `variable` blocks
- `outputs.tf` — all `output` blocks

**Update the `terraform {}` block** in `main.tf` — replace the no-backend comment
with the HCP Terraform `cloud` block:

```hcl
terraform {
  required_version = ">= 1.6"

  cloud {
    organization = "<your-hcp-org-name>"
    workspaces {
      name = "google-calendar-sa"
    }
  }

  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}
```

## Phase 4 — Migrate state from OpenTofu local → HCP Terraform

The existing state records the provider source as
`registry.opentofu.org/hashicorp/google`. HashiCorp Terraform uses
`registry.terraform.io`. The state must be patched before Terraform will
accept it.

### 4a. Copy the state file locally (do NOT commit it)

```powershell
Copy-Item `
  "<path-to-adk-playground>\terraform\modules\google-calendar-sa\terraform.tfstate" `
  "terraform-adk-agents\terraform.tfstate"
```

### 4b. Patch the provider registry URL

In `terraform-adk-agents/terraform.tfstate`, find-replace:

- **Find:** `registry.opentofu.org/hashicorp/google`
- **Replace:** `registry.terraform.io/hashicorp/google`

### 4c. Authenticate to HCP Terraform

```bash
terraform login
```

### 4d. Initialize — Terraform will offer to migrate state

```bash
cd terraform-adk-agents
terraform init
```

Terraform detects the local `terraform.tfstate` next to the `cloud {}` backend
and prompts:

```
Do you want to copy existing state to the new backend?
  Enter a value: yes
```

State is now stored in HCP Terraform. The local file is no longer needed.

### 4e. Verify, then delete the local state file

```bash
terraform state list   # expect: 3 resources
terraform plan         # expect: No changes
```

```powershell
Remove-Item terraform-adk-agents\terraform.tfstate
```

## Phase 5 — Configure variables in HCP Terraform

Never commit real variable values to the public repo. Set them in the HCP
Terraform workspace UI under **Variables**:

| Variable | Sensitive | Category | Notes |
|---|---|---|---|
| `project_id` | No | Terraform | GCP project ID |
| `GOOGLE_CREDENTIALS` | **Yes** | Environment | Full JSON of a GCP service account key with permission to manage IAM in the project |

`GOOGLE_CREDENTIALS` is the Google provider's standard env var for CI
authentication — no key file on disk or in code required.

Add a `terraform.tfvars.example` in the module directory to document what
variables callers must supply:

```hcl
# Copy to terraform.tfvars (gitignored) for local runs.
# In HCP Terraform, set these as workspace variables instead.
project_id = "your-gcp-project-id"
# region = "us-central1"  # optional, defaults to us-central1
```

## Phase 6 — VCS integration (HCP Terraform ↔ GitHub)

Link the HCP Terraform workspace to this repo so plans and applies are
driven by pull requests — no CI secrets needed in GitHub.

1. HCP Terraform → Workspace → **Settings → Version Control**
2. Connect to GitHub → select `matthewshan/cloud-infrastructure`
3. Set **Working Directory** to `terraform-adk-agents/`
4. Configure run triggers:
   - **Speculative plan** on every PR targeting `main`
   - **Apply** on merge to `main`
